# ADR 0002 — PDF ingestion and structured extraction

## Status

Accepted (2026-07-08)

## Context

Through M5 the corpus was hand-authored Markdown: `ingestCompany` read each file
with `readFile(…, 'utf8')`, chunked, embedded, and stored it. That demonstrates
retrieval, the agent, and the eval loop — but it skips the _front half_ of a real
system: turning an unstructured filing (a PDF) into text, and then into typed,
DD-relevant fields. That front half is the most direct expression of the author's
prior production experience (an LLM extraction pipeline: PDF → structured fields
via Claude, with an LLM-as-judge quality gate). M6 adds it, on this project's own
domain (company filings) and within its constraints: keyless-by-default CI, the
provider-switch pattern, a tiny corpus, mature SDKs over bespoke plumbing.

## Decision

- **PDF decode via `unpdf` (default, keyless).** `unpdf` is a serverless build of
  pdf.js — pure-JS, ESM-native, no native deps, no key — so it decodes text-layer
  PDFs inside the keyless demo/eval/CI path. `decodePdf` (`src/pdf.ts`) is the
  seam; the ingest loader dispatches on file extension (`.md`/`.txt` verbatim,
  `.pdf` → decode), so the existing Markdown path is unchanged.
- **Amazon Textract is a documented swap, not built.** OCR / scanned-image /
  multi-column filings are Textract's domain; `PDF_PROVIDER=textract` is a guarded
  seam that points here rather than shipping AWS plumbing that can't run keyless.
  This mirrors ADR 0001's Railway-vs-AWS posture (a documented target, not
  operated). Scope here is deliberately **text-layer only**.
- **Structured extraction with a provider switch, defaulting to local.**
  `extract` (`src/extract.ts`) returns typed fields (a Zod schema whose four
  groups map 1:1 onto the four checks), each with an evidence snippet. Like the
  reasoner it has an `anthropic` path (`generateObject` + the schema) and a
  keyless `local` path (deterministic regexes reusing the reasoner's
  negation-aware signals). Unlike the reasoner, the **default is `local`**:
  extraction runs inside the _ingest_ path that the keyless demo/eval/CI all
  exercise, so a forgotten env var must fall back to keyless — the same inversion
  `judge.ts` uses.
- **Persist on the document, as one `jsonb` column.** `documents.extraction`
  (nullable) stores each document's fields — no new table or join at this corpus
  size. Surfaced via `GET /extract/:company` and the demo.
- **Prove it with a keyless F1 test.** `test/extract.test.ts` decodes each PDF
  fixture, extracts with the local provider, and scores field-level
  precision/recall/**F1** against a golden set (`src/golden-extract.ts`), echoing
  the real pipeline's Recall/F1 framing. It rides `npm test` — no new CI step, no
  DB, no key. An extraction-specific LLM-as-judge is documented, not built (the F1
  test is the proof; the report path already carries a real LLM-as-judge in M4).

## Consequences

- One new runtime dependency (`unpdf`); everything else stays keyless and
  dependency-light. The PDF fixtures are committed binaries generated by a
  committed, reproducible script (`npm run fixtures:pdf`) so the corpus stays
  walkable cold; CI never renders them.
- The PDF fixtures deliberately restate each company's planted signals, so adding
  them to the retrievable corpus **reinforces** the golden verdicts rather than
  perturbing them — the M4 eval stays 12/12.
- Explicitly out of scope (kept lean, and noted so the boundary is deliberate):
  OCR / scanned PDFs, multi-column layout reconstruction (brochure-specific, low
  ROI for single-column filings), a built Textract provider, and DSPy-style prompt
  optimisation (Python — off-language for this TS/Node project).
